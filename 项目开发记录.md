# 视障助行多模态项目开发记录

## 一、项目背景与目标
本项目面向云计算课程大作业，目标是实现一个可落地演示的视障助行多模态原型：
1. 前端支持图片上传、视频上传、实时拍摄三种输入方式。
2. 后端通过云函数接收 OSS 图片地址，调用视觉大模型生成路况描述，再调用语音能力返回音频。
3. 在试用额度约束下，优先保证系统可用性、稳定性与演示体验。

## 二、阶段性开发历程
### 阶段 1：基础后端能力落地
1. 建立 Flask Web 函数接口，兼容 POST / 与 POST /api/analyze。
2. 增加全局 CORS 支持，处理 OPTIONS 预检，确保本地前端可跨域调用。
3. 接入视觉分析与语音合成，并按统一 JSON 协议返回 text + audioBase64。
4. 完成异常兜底，避免函数崩溃导致演示中断。

阶段收获：
- 云函数环境最关键的是“稳定返回”，包括请求格式校验、第三方调用失败处理、CORS 一次性处理到位。

### 阶段 2：前端主链路打通（上传图片）
1. 初始化 React + Vite 项目并接入 Ant Design。
2. 使用 Zustand 建立统一状态机：idle、uploading、analyzing、success、error。
3. 完成浏览器文件到 OSS 的直传，再将 OSS URL 提交后端分析。
4. 结果展示区实现文本展示与音频播放。

阶段收获：
- 采用 Zustand 管理跨组件异步流，明显降低了页面状态错乱概率。
- 先打通单图主链路，便于快速发现后端接口与前端数据结构不一致问题。

### 阶段 3：UI 与移动端体验修复
1. 发现上传区“内层虚线框与底部背景框错位”问题。
2. 修复方式：约束上传容器宽高、统一盒模型、补齐溢出裁剪、覆盖 Dragger 内部样式。
3. 回归验证后，移动端视觉结构恢复稳定。

阶段收获：
- 组件库默认样式在移动端经常出现边界行为，关键是缩小布局自由度并控制溢出区域。

### 阶段 4：工程化能力完善（Git）
1. 初始化仓库并建立主分支 main。
2. 完善 .gitignore，覆盖 Node、Python、IDE、日志、环境变量等场景。
3. 绑定远程仓库并完成首次推送。

阶段收获：
- 早做仓库规范可以避免敏感配置误提交，减少后续整理成本。

### 阶段 5：能力扩展（上传视频 + 实时拍摄）
1. 上传模式扩展为双模式：上传图片、上传视频。
2. 新增实时拍摄模块：打开摄像头、按间隔抽帧、单轮上限控制、暂停与关闭控制。
3. 额度护栏加入状态层：会话总分析上限、实时抽帧间隔下限、单轮帧数上限。

阶段收获：
- 在试用资源限制下，“可控地慢一点”比“无约束地快一点”更适合课程项目演示。

### 阶段 6：语音防抖与视频自适应抽帧
1. 语音防抖：实时模式下，对新描述与上一条播报文本做相似度判断。
2. 当差异不明显时，跳过自动语音播报，减少重复提醒和资源消耗。
3. 视频抽帧升级为“按时长自适应”：短视频少帧、长视频多帧，并受额度和上限约束。

阶段收获：
- 防抖不仅是节省资源，更能提升听感质量，避免“同一句话反复播报”的负体验。
- 自适应抽帧是准确率与成本的平衡点，优于固定三帧或逐帧分析。

## 三、关键技术决策与理由
### 1. 为什么采用 OSS 直传
1. 前端直接上传 OSS，减少后端承载文件流的压力。
2. 后端只处理 URL，接口更轻、更符合云函数短时执行模型。
3. 便于后续切换到视频关键帧或实时抓帧等多输入来源。

### 2. 为什么用 Zustand 作为状态核心
1. 上传、分析、播放是跨组件多步骤异步流。
2. Zustand 让所有状态切换可追踪，调试成本低。
3. 能自然接入额度控制、来源标记、防抖标记等全局策略。

### 3. 为什么实时模式采用抽帧而非全视频流
1. 试用资源条件下，全流实时分析成本高且不稳定。
2. 周期抽帧更可控，便于讲清楚系统策略和节流逻辑。
3. 对课程项目演示场景，识别间隔在可接受范围内即可体现系统价值。

### 4. 为什么语音防抖放在状态层而不是组件层
1. 防抖属于业务策略，应该与数据流同层管理。
2. 组件只负责展示与播放，不应承载复杂策略判断。
3. 后续若接入更多输入源，可复用统一防抖逻辑。

## 四、遇到的典型问题与解决方案
### 问题 1：上传区布局错位（移动端）
现象：上传虚线框底边与卡片底边错位，视觉上有“嵌套漂移”。

排查结论：
1. 组件默认样式叠加导致内部元素高度与父容器约束不一致。
2. 外层容器未裁剪内部溢出，圆角边界表现异常。

解决方案：
1. 统一上传容器宽高占比。
2. 强制内层内容区域对齐。
3. 开启父容器溢出裁剪。

### 问题 2：实时模式重复播报影响体验
现象：画面变化不大时，系统反复播报相似文本。

解决方案：
1. 文本归一化后进行二元组相似度计算。
2. 设置可调阈值，差异不足时跳过自动播放。
3. 保留文本展示与手动播放入口，避免信息丢失。

### 问题 3：视频固定三帧信息不足
现象：中长视频场景覆盖不够，可能遗漏关键路况。

解决方案：
1. 按视频时长推荐抽帧数量。
2. 在时间轴均匀采样关键点。
3. 增加上限和额度剩余约束，兼顾效果与成本。

### 问题 4：Git 工作流初始化较晚
现象：在功能快速迭代后才开始仓库管理，增加了确认成本。

改进：
1. 补齐 .gitignore 并审查敏感文件。
2. 分批提交并记录功能演进。
3. 与远程仓库绑定后及时推送，降低本地丢失风险。

## 五、可用于答辩展示的设计巧思
1. 成本感知策略
- 会话分析上限
- 实时抽帧间隔下限
- 单轮最大帧数
- 视频时长自适应抽帧
- 语音防抖减少重复播报

2. 用户体验策略
- 三输入模式并存：图片、视频、实时拍摄
- 状态机清晰可视化：上传中、分析中、成功、失败
- 自动播放失败时保留手动播放兜底

3. 工程可维护策略
- 状态层统一承载业务规则
- 组件职责聚焦展示和交互
- 关键配置环境变量化，便于演示前快速调参

## 六、项目中的主要心得体会
### 1. 技术实现不应脱离资源约束
本项目最重要的认知变化是：设计要从“资源现实”出发，而不是只从“功能理想”出发。试用额度受限时，系统策略的意义远大于单点算法炫技。

### 2. 云原生系统的核心不是上云，而是解耦
前端负责采集与上传，OSS负责存储，云函数负责编排模型调用，职责分离后每一层都更容易定位问题，也更适合课程项目的阶段性交付。

### 3. 演示系统需要把失败路径也设计好
真正能打动评审的不是“永远成功”，而是“失败也可控”：
1. 参数缺失有明确报错
2. 网络异常有友好提示
3. 自动播放受限仍可手动播放
4. 达到额度上限有可解释反馈

### 4. 状态机思维提升了开发效率
把复杂异步流程变成可观测状态机后，排查问题时可以快速定位在哪个阶段出错，避免“黑盒式调试”。

## 七、当前代码能力边界
1. 实时模式本质是“定时抓帧分析”，不是 WebRTC 全双工会话。
2. 语音播放仍是单次返回后播放，不是边合成边播流式音频。
3. 视频分析依赖关键帧采样，不能替代真正的视频时序模型。

## 八、后续可迭代方向
1. 引入更细粒度的变化检测，再决定是否上传抓帧（先本地筛帧再云端分析）。
2. 增加播报队列与打断控制，减少重叠播放。
3. 将配置参数做成页面可调面板，提升答辩演示可解释性。
4. 增加最小化监控统计，如每轮调用次数、平均响应时延、播报命中率。

## 十、总结
本项目最终形成了一个“可演示、可解释、可迭代”的云原生多模态原型。相比单一功能堆叠，系统真正的价值体现在三点：
1. 全链路打通：从采集到播报闭环可运行。
2. 成本可控：在试用资源下也能稳定演示。
3. 结构清晰：便于继续演进到更强的实时交互形态。
